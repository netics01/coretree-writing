# Coretree Writing Specification

이 문서는 `SPEC.md`의 한국어 companion 문서입니다. 영어 `SPEC.md`를 우선합니다.

이 문서는 Coretree 작성 스타일을 한국어로 함께 설명합니다.

## 정의

Coretree는 정보를 읽기 쉬운 멀티레벨 불릿으로 조직하는 작성 스타일입니다.

- 정보를 짧은 단위로 압축합니다.
- 정보 사이의 관계를 드러냅니다.
- 불릿 깊이로 종속성을 표현합니다.
- 위치, 순서, 라벨로 중요도를 표현합니다.
- 사람이 빠르게 훑을 수 있도록 위계를 얕고 명확하게 유지합니다.

## 원칙

- 긴 문단보다 불릿을 우선합니다.
- 한 불릿에는 하나의 정보만 둡니다.
- 상위 불릿에는 핵심 주제, 결정, 결과, 분류를 둡니다.
- 하위 불릿에는 세부 내용, 원인, 조건, 근거, 예시, 후속 정보를 둡니다.
- 같은 깊이의 불릿은 같은 성격의 정보로 맞춥니다.
- 반복되거나 가치가 낮은 표현은 압축합니다.
- 구어체 흐름보다 짧은 정보 단위를 우선합니다.
- 라벨은 훑어보기에 도움이 될 때 사용하며, 모든 불릿에 필수는 아닙니다.

## 불릿 깊이

깊이는 의도적으로 사용합니다.

- 1단계: 핵심 주제
- 2단계: 핵심 주제에 대한 주요 정보
- 3단계: 필요한 세부 정보
- 4단계: 드물게, 명확성이 좋아질 때만 사용
- 5단계: 피함

Markdown 불릿의 하위 단계는 단계마다 4칸 공백으로 들여씁니다. 탭이나 2칸 들여쓰기는 사용하지 않습니다. 들여쓰기 폭을 일정하게 유지해 위계가 명확히 보이도록 합니다.

원래 정보가 깊다고 해서 깊은 구조를 그대로 보존하지 않습니다. 트리가 너무 깊어지면 재조직합니다.

- 정보를 별도 주제로 나눕니다.
- 비슷한 세부 정보를 합칩니다.
- 불필요한 중간 라벨은 이름을 바꾸거나 제거합니다.
- 짧은 위계는 한 줄로 합칩니다.
- 중요한 정보는 맥락을 붙여 상위로 올립니다.

## 중요도와 종속성

불릿 깊이는 종속성을 보여줍니다. 낮은 중요도를 의미하지 않습니다.

중요한 정보는 깊은 하위 불릿에 숨기지 않습니다. 중요한 세부 정보라면 상위로 올리고 충분한 맥락을 붙입니다.

약한 형태:

```text
- 공급업체 평가
    - 도구 A
        - 한국어 정확도 부족
        - 판정: 탈락
```

선호 형태:

```text
- 도구 A 판정: 탈락
- 탈락 근거: 한국어 정확도 부족
```

## 라벨

라벨은 정보의 역할을 더 쉽게 훑어보게 만들 때 사용합니다.

라벨은 선택 사항입니다. 불릿의 역할이 문맥만으로 분명하면 모든 줄을 `Label: value` 형태로 강제하기보다 자연어 불릿이 더 나을 수 있습니다.

- `Decision:`: 선택된 결과
- `Result:`: 발생한 결과
- `Cause:`: 원인
- `Status:`: 현재 상태
- `Risk:`: 가능한 문제
- `Next:`: 후속 행동

라벨은 짧아야 합니다. 라벨만 봐도 주변 문장을 다시 읽지 않고 정보를 훑을 수 있어야 합니다.

약한 형태:

```text
- 온보딩 흐름 논의
    - 문제: 환영 화면이 너무 김
    - 사용자 행동: 설정 단계를 자주 건너뜀
    - 결정: 환영 문구 축소
    - 다음: 설정 안내를 첫 작업 화면으로 이동
```

선호 형태:

```text
- 온보딩 흐름 결정: 환영 문구 축소
    - 환영 화면이 너무 김
    - 사용자가 설정 단계를 자주 건너뜀
    - 설정 안내를 첫 작업 화면으로 이동
```

## 한 줄 관계

짧은 관계는 한 줄로 합칠 수 있습니다.

`:`는 라벨-값 관계에 사용합니다.

```text
- Decision: rejected
- Status: blocked
- Cause: missing permission
```

`=>`는 원인, 변화, 결과에 사용합니다.

```text
- Missing permission => input failed in other apps
- More transcription errors => higher review cost
```

한 줄이 더 명확한 관계를 굳이 새 불릿 단계로 만들지 않습니다.

약한 형태:

```text
- 판정
    - 탈락
```

선호 형태:

```text
- 판정: 탈락
```

## 사용할 때

Coretree는 정보를 빠르게 훑거나, 나중에 다시 보거나, 이어서 확장해야 할 때 사용합니다.

- 회의록
- 의사결정 기록
- 리서치 요약
- 버그 리포트
- 프로젝트 상태 업데이트
- AI 에이전트 출력
- 세부 설명이 붙은 체크리스트

## 쓰지 않을 때

산문이 더 적절한 곳에 Coretree를 강제로 적용하지 않습니다.

- 리듬이나 목소리가 중요한 서사형 글
- 설득형 에세이
- 정확한 문구가 중요한 법률/정책 문서
- 위계가 없는 짧은 메시지
- 표, 다이어그램, 코드 블록이 더 명확한 내용

## 작성 체크리스트

- 결론이나 결정이 상단에 보이는가?
- 각 불릿에 하나의 정보만 있는가?
- 깊이가 중요도가 아니라 종속성을 나타내는가?
- 하위 Markdown 불릿을 단계마다 4칸 공백으로 들여썼는가?
- 라벨이 짧고 유용한가?
- 4단계 불릿이 드물게만 쓰였는가?
- 5단계 불릿을 피했는가?
- 짧은 위계를 `:` 또는 `=>` 한 줄로 바꿀 수 있는가?
